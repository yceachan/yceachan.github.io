# RTOS 的第一性原理：确定性的时间切片

RTOS (Real-Time Operating System) 的本质不是“快”，而是 **“确定性 (Determinism)”**。
它的核心任务是：**在极其有限的资源约束下，提供可预测的时间响应。**

---

## 1. 灵魂：确定性 (Determinism)

通用操作系统（如 Linux/Windows）追求的是 **吞吐率 (Throughput)** —— 单位时间内处理尽可能多的任务。
RTOS 追求的是 **延迟 (Latency)** —— 在规定的截止时间 (Deadline) 前必须完成任务。

### 1.1 第一性原理：优先级抢占 (Preemptive Scheduling)
*   **铁律：** 任何时刻，**CPU 必须运行优先级最高的就绪任务**。
*   **机制：** 一旦高优先级中断发生（如 UART 接收到数据），内核必须立即暂停当前低优先级任务，切换上下文。
*   **指标：** **中断延迟 (Interrupt Latency)** 和 **调度延迟 (Scheduling Latency)** 必须是常数级（O(1)），与系统负载无关。

### 1.2 优先级反转 (Priority Inversion) —— 确定性的天敌
*   **现象：** 高优先级任务被低优先级任务阻塞（因为低优先级任务占用了互斥锁），而中优先级任务又打断了低优先级任务。结果高优先级任务反而最后执行。
*   **解法：** **优先级继承 (Priority Inheritance)**。当低优先级任务持有高优先级任务需要的锁时，临时提升其优先级，直到释放锁。

---

## 2. 肉体：上下文切换 (Context Switch)

RTOS 是如何“同时”运行多个任务的？其实是假的。
它利用 CPU 的速度极快，快速地在任务之间切换，制造出并行的假象。

### 2.1 第一性原理：栈 (Stack) 与 寄存器 (Registers)
*   **任务 (Task) 的本质：** 一个独立的函数 + **私有的栈空间 (Stack)** + **一组寄存器快照 (TCB)**。
*   **切换过程 (The Magic Trick):**
    1.  **保存现场 (Save Context):** 触发 PendSV 中断，将当前 CPU 所有的通用寄存器 (R0-R15, PSR) 压入当前任务的栈中。
    2.  **切换栈指针 (Switch SP):** 将 CPU 的栈指针 (SP) 指向下一个任务的栈顶。
    3.  **恢复现场 (Restore Context):** 从新栈中弹出寄存器值，恢复到 CPU 中。
    4.  **跳转 (Return):** 执行 `BX LR`，CPU “以为”自己刚从上一个中断返回，实际上已经到了另一个函数的中间。

### 2.2 核心组件：SysTick (系统滴答)
*   **心跳：** RTOS 需要一个硬件定时器产生周期性中断（通常 1ms）。
*   **作用：** 唤醒延时任务 (`vTaskDelay`)，检查时间片轮转 (Round Robin)。

---

## 3. 通信：任务间同步 (IPC)

RTOS 的另一个核心职责是管理任务间的协作。

*   **信号量 (Semaphore):** "我做完了，你可以开始了。" (同步)
*   **互斥锁 (Mutex):** "我在用厕所，你排队。" (资源保护)
*   **消息队列 (Queue):** "这是数据，拿去处理。" (数据传输)

**第一性原理：** 所有的 IPC 机制，本质上都是 **“阻塞 (Blocking)”** 与 **“唤醒 (Unblocking)”** 的操作。
*   当资源不可用时，任务主动让出 CPU（进入 Blocked 态），而不是死循环轮询（浪费 CPU）。

---

## 总结

RTOS = **一个极简的调度器核心 (Scheduler)** + **一群围绕确定性服务的对象 (Tasks)**。

*   **对内 (CPU):** 利用 **中断 (PendSV)** 和 **栈 (Stack)** 玩弄时间，实现多任务的幻觉。
*   **对外 (App):** 提供 **严格的时序保证 (Hard Real-Time)**，确保关键任务永不掉链子。
