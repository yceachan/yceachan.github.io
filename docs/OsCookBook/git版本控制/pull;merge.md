- # Git 合并与拉取全景指南

  ## 1. 核心合并策略对比

  ### 1.1 Fast-forward (默认)

  当合并的分支是当前分支的直接后代时，Git 默认只移动指针。

  - **优点**：历史线简洁，没有额外的合并节点。
  - **缺点**：丢失了“特性分支”存在的历史痕迹。

  ### 1.2 `--no-ff` (强制生成合并节点)

  即便可以快进合并，也强制创建一个新的合并提交。

  - **意义**：清晰标示出某次功能合入的起始与终点，方便回滚（Revert）。

  ### 1.3 `git pull --rebase` (变基拉取)

  将本地未推送的提交“移到”远程最新提交之后。

  - **意义**：保持主干历史呈一条直线，避免产生大量 "Merge branch 'main' of..." 的垃圾节点。

  ---

  ## 2. 参数全字典 (Args Dictionary)

  ### git merge [args]

  - **`--ff` (默认)**：尽可能快进。
  - **`--no-ff`**：禁用快进，始终创建合并提交。
  - **`--ff-only`**：如果不能快进，直接报错（防止产生不必要的合并线）。
  - **`--squash`**：将多个提交压缩为一个，放在暂存区，不自动生成提交。
  - **`--abort`**：发生冲突时，彻底撤回合并操作。

  ### git pull [args]

  - **`(无参数)`**：等同于 `fetch` + `merge`。
  - **`--rebase`**：等同于 `fetch` + `rebase`。
  - **`--ff-only`**：拉取时如果本地有变动无法快进，则报错。
  - **`--autostash`**：拉取前自动 stash 本地未提交的改动，rebase 结束后自动 pop。

  ---

  ## 3. 最佳实践流程

  1. **本地开发时**：频繁使用 `git pull --rebase` 同步远程。
  2. **特性合入主干时**：使用 `git merge --no-ff feature-name`。
  3. **推送到远程前**：检查 `git log --oneline --graph`，确保历史清晰。
