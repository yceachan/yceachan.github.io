# 2026-04-20 站点改版

> [!note]
> **Ref:** 本次改动涉及 `docs/.vitepress/config.mts`、`docs/.vitepress/theme/style.css`、`docs/.vitepress/theme/index.ts`、`docs/.vitepress/plugins/docTreePlugin.ts`、以及 `components/` 下的 `Layout.vue`、`ExplorerList.vue`、`ExplorerBreadcrumb.vue`。

## 一、侧边栏 (Sidebar) 相关

### 1.1 去重：移除 vitepress-sidebar 自动生成

**现象**：`config.mts` 里同时跑了 `generateSidebar(...)`（第三方包）和自写的 `walkDocsTree(...)`，前者的产物仅被挂到 `scopedSidebarConfig['/']`，而 `/` 路径（首页 Explorer）又被 `SidebarProfileWrapper` 完全覆盖 —— 所以 `generateSidebar` 的结果**永远不会被用户看到**。

**处理**：

- 删除 `import { generateSidebar }` 与 `sidebarConfig` 变量
- `scopedSidebarConfig` 初始化为 `{}`，只由 `collectScopedSidebar` 逐目录填入

### 1.2 修复：短屏下文件树顶穿标题

**现象**：在 iPad 横屏（高 768）这类 "宽但矮" 的设备上，滚动侧栏会把 item 内容顶到顶部，覆盖 `yceachan's Knowledge Base` 这行站点 title。

**根因**：我们在 `style.css` 里给 `.VPSidebar` 写了：

```css
.VPSidebar {
  display: flex !important;
  flex-direction: column !important;
  ...
}
```

而 VitePress 默认结构里有个 `.curtain` 子元素，靠 `position: sticky; top: -64px` 在顶部挡住滚动上溢。`flex` 容器会改变 sticky 的生效条件，导致 curtain 失效，items 直接穿越顶部。

**处理**：把 flex 布局从 `.VPSidebar` 移到 `.VPSidebar > .nav`，并用 `min-height: 100%` 填满，`sidebar-nav-after` 注入的 Copyright 仍可靠 `margin-top: auto` 贴底。

### 1.3 文件树换行与间距

长文件名在窄侧栏下常被截断，现在允许换行，同时收紧了单项纵向间距：

```css
.VPSidebarItem .text {
  white-space: normal !important;
  line-height: 1.45;
  padding: 3px 0;
}
.VPSidebarItem .item {
  padding-top: 2px;
  padding-bottom: 2px;
}
```

### 1.4 移动端 padding 压缩

VitePress 默认移动端 `.VPSidebar` 是 `padding: 32px 32px 96px` —— 竖屏手机上左右各吃掉 32px 过于浪费：

```css
@media (max-width: 959px) {
  .VPSidebar {
    padding-left: 12px !important;
    padding-right: 12px !important;
    padding-top: 8px !important;
  }
}
```

> [!warning]
> 注意 **不要** 把 `padding-top` 写成 `var(--vp-nav-height) + 8`。移动端 `.VPSidebar` 默认 padding-top 只有 32px（curtain 仅 `min-width: 960px` 才 sticky），加 `var(--vp-nav-height)` 反而会让顶部空间更大。

## 二、文件树命名：文件名而非 H1

**现象**：Explorer 与 VPSidebar 都扫描 `.md` 的首个 `# H1` 作为显示名，导致两处显示和 URL slug 对不上。

**处理**：

- `docs/.vitepress/plugins/docTreePlugin.ts` 删除 `readH1()`，直接用 `item.replace(/\.md$/, '')`
- `docs/.vitepress/config.mts` 中 `walkDocsTree()` 同步删除 `readH1Title()`

## 三、Explorer 排序：英文在前，中文在后

原先 `localeCompare(..., 'zh-Hans-CN')` 会按拼音混排中英文。改为先按首字符 ASCII 范围分组：

```ts
const aAscii = a.name.charCodeAt(0) < 0x80
const bAscii = b.name.charCodeAt(0) < 0x80
if (aAscii !== bAscii) return aAscii ? -1 : 1
return a.name.localeCompare(b.name, 'zh-Hans-CN')
```

Explorer 和 VPSidebar 使用同一套排序规则保持一致。

## 四、面包屑：在 Markdown 页面也显示

把 `ExplorerBreadcrumb` 通过 `doc-before` slot 注入到文档页：

- Explorer 页：路径来自 `explorerStore.currentPath`（URL query `?path=`）
- Markdown 页：路径来自 `route.path`，最后一段（文件名）标记为 `isCurrent`，不可点击

点击中间段会跳回 Explorer `/?path=...`。

## 五、正文列宽 (PC)

**现象**：PC 三栏布局下（Sidebar / Content / On this page），中间正文太窄。

**排查**：不是 padding 引起的。`node_modules/vitepress/dist/client/theme-default/components/VPDoc.vue:190` 写死了：

```css
.VPDoc.has-aside .content-container {
  max-width: 688px;
}
```

这个 688px 硬盖无论视口多宽都限死正文宽度。

**处理**：

```css
.VPDoc.has-aside .content-container {
  max-width: none !important;
}
```

## 六、GitHub 风格 Alert 背景高亮

**现象**：`> [!NOTE]` / `> [!TIP]` 原先是灰底 + 灰左框（`#8E96AA24`），辨识度低。

**排查陷阱**：`theme/index.ts` 里 import 了 `markdown-it-github-alerts` 的 CSS（`.markdown-alert.markdown-alert-note` 命名），但本项目**没注册**该 markdown-it 插件 —— VitePress 1.5+ 自带了 `> [!NOTE]` 解析，输出的是 `<div class="note custom-block github-alert">`。所以先前那套 `.markdown-alert` 规则永远无法匹配。

**处理**：给 `.vp-doc .custom-block.note.github-alert` 等真实类名加 10% 色底（暗色 15%）+ 同色左边框 + 标题色：

| 类型 | 颜色 |
| --- | --- |
| note | `#0969da` 蓝 |
| tip | `#1a7f37` 绿 |
| warning | `#9a6700` 黄 |
| important | `#8250df` 紫 |
| caution | `#d1242f` 红 |

## 七、小结

这次整理的核心教训：

1. **排查 CSS 覆盖问题时，先打开 DevTools 看实际类名**，不要照搬插件文档 —— VitePress 自带的 `[!NOTE]` 渲染用的是自己的 `.custom-block` 命名，和 `markdown-it-github-alerts` 包不同。
2. **改 `.VPSidebar` 的 display 属性要谨慎**：它里面有 `.curtain` 依赖 sticky 的结构，改成 flex 会静默破坏短屏滚动体验。
3. **宽度被硬 cap 住时，先查 VitePress 的 scoped `.vue` 样式**，不是所有约束都写在 CSS 变量里。
