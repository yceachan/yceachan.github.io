# 2026-04-22 Explorer 导航修复

> [!note]
> **Ref:** 涉及 `docs/.vitepress/components/ExplorerItem.vue`、`docs/.vitepress/components/ExplorerBreadcrumb.vue`；参考 `node_modules/vitepress/dist/client/app/router.js` 与 `docs/.vitepress/config.mts` 的 VitePWA 配置。

## 一、现象

在 Explorer（首页 `/`）内点击目录图标或面包屑跳转：

- URL 的 `?path=X` **已经**更新
- 当前视图**仍停留**在原目录列表
- 按 F5 整页刷新后才对上

关键的环境差异 —— **dev 不复现，只有 prod 才观察到**：

- `vitepress dev` 下无法复现
- GitHub Pages 线上站点 / 本地 `vitepress build` + `npx serve` 复现
- 服务器日志大量 `304 Not Modified`（哈希命名 chunk 的浏览器再验证，**不是** bug 本体，只是 prod 走真实网络的信号）

## 二、根因

Explorer 内 `router.go('/?path=X')` 的 URL 与当前路由 pathname 一致（都是 `/`），`explorerStore.currentPath` 的刷新完全依赖 `Explorer.vue` 在 `<script setup>` 顶层注册的 `router.onAfterRouteChange` 钩子：

```js
// node_modules/vitepress/dist/client/app/router.js
async function go(href) {
  ...
  await loadPage(href)                          // prod 下这是真实网络 fetch
  await router.onAfterRouteChange?.(href)       // 回来再同步 store
}
```

**dev**：`loadPage` 在 Vite ESM 内存里同步解析，钩子在同一 tick 触发，无感知。

**prod**：

1. `history.pushState` 已把 URL 改成 `/?path=X`（同步完成）—— 用户立刻看到 URL 变化
2. `await loadPage(href)` 进入真实网络获取；叠加 `vite-plugin-pwa` / workbox 预缓存（见 `docs/.vitepress/config.mts:176` 的 `VitePWA({...})`）service worker 拦截，返回的可能是同一模块引用，`route.component = markRaw(comp)` 赋同引用不触发 Vue 再渲染
3. 此间隙内 `explorerStore.currentPath` 仍是旧值，`ExplorerList` 的 computed 没有新输入
4. `Explorer.vue` 在 `<script setup>` 顶层对 `router.onAfterRouteChange` 做了 `const prevAfter = router.onAfterRouteChange; router.onAfterRouteChange = wrap` 的包装/还原，生命周期（从子页面返回 Explorer、SW 冷启动预缓存）稍一错位这条链就断，回调根本不来

F5 能修好，是因为整页刷新走 `Explorer.vue` 的 `onMounted → syncPathFromUrl`，这条路径不依赖异步钩子。

> [!warning]
> `304 Not Modified` 是 prod 构建的正常结果（哈希命名 chunk + `If-None-Match`），它只是"真实网络发生"的指示灯，不代表服务器/缓存故障。别被日志吓到去追缓存配置。

## 三、处理

**思路**：让视图刷新**不再依赖** `router.onAfterRouteChange` 在 prod 下按时触发。点击时直接同步改 `explorerStore.currentPath`，`router.go` 只管改 URL + 加载模块。

### 3.1 `components/ExplorerItem.vue`

```ts
function handleClick() {
  if (props.node.type === 'dir') {
    // 同步更新 store，避免仅依赖 router.onAfterRouteChange —
    // 同路由下仅 query 变化时该钩子偶发不触发，导致 URL 已改但视图不刷新。
    explorerStore.currentPath = props.node.path
    router.go(`/?path=${encodeURIComponent(props.node.path)}`)
  } else {
    router.go(withBase(props.node.path))
  }
}
```

### 3.2 `components/ExplorerBreadcrumb.vue`

面包屑在 Explorer 页（`context='explorer'`）与 doc 页（`doc-before` slot）都会渲染，只有 Explorer 内跳转属于"同路由仅 query 变"的 bug 路径，因此用 `isExplorer.value` 守卫：

```ts
function go(path: string) {
  if (isExplorer.value) explorerStore.currentPath = path
  router.go(`/?path=${encodeURIComponent(path)}`)
}
```

doc 页上点面包屑是跨路由导航，Explorer 挂载时 `onMounted → syncPathFromUrl` 会从 URL 读取 `path` 写入 store，该路径一直可靠，无需改。

### 3.3 保留的兜底

- `Explorer.vue` 现有的 `onAfterRouteChange` 包装、`popstate` 监听、`onMounted` 同步全部保留，作为浏览器前进/后退 + 整页刷新的兜底
- `NavLeftActions.vue`、`BackToExplorerButton.vue` 不改 —— 它们都是 doc 页 → Explorer 的跨路由导航，走 `onMounted` 路径，不涉及本 bug

## 四、验证

本 bug dev 模式本就不复现，**必须在 prod 产物下验证**：

```bash
npm run docs:build
npx serve docs/.vitepress/dist -l 4173
```

用隐身窗口打开 `http://localhost:4173/` 避开 SW 旧缓存；或 DevTools → Application → Service Workers → Unregister 后再测。

测试矩阵（每次**单击**就应即时刷新，不需 F5）：

- 首页点根目录图标 → 立即显示该目录内容
- 面包屑连续点击多级父目录 → 每次都立即刷新
- 浏览器"后退/前进" → 视图同步（`popstate` 兜底）
- 子目录 URL 下按 F5 → 视图对上（`onMounted` 兜底）
- doc 页点 Nav 的 Back/Home → 返回 Explorer 视图对上

Network 面板：每次点击仍会有 `index.*.js` 请求（大概率 304），但视图刷新**不依赖**该请求完成 —— 点击瞬间 `ExplorerList` 就应换内容。

## 五、小结

核心教训：

1. **dev 正常不代表 prod 正常** —— Vite ESM 内存态把很多依赖"同 tick 异步回调"的代码伪装成同步可用；prod 的真实网络 + SW 拦截一旦引入异步间隙，隐藏问题才显形。
2. **UI 状态不要只由路由钩子驱动** —— 点击时直接改响应式 store 最可靠；路由钩子作为 URL→state 的**回同步**兜底，而不是主通道。
3. **304 不是 bug**，只是你现在走了真实网络的证据，先分清信号和噪音再追问题。
