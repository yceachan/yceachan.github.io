# xv6 物理内存管理（kalloc）

> 从硬件到软件，完整理解 xv6 如何管理物理内存的分配与释放。

---

## 1. 硬件基础：内存到底是什么

### 1.1 内存的物理结构

内存（RAM）在物理上是一个巨大的**二维网格**，每个格子存 1 bit（0 或 1）。硬件把每连续 8 个格子编为一组，称为 1 **字节**（byte）。每个字节有一个唯一编号，即**地址**。

所以从编程的角度看，内存就是一个**巨大的字节数组**，所谓"页"只是操作系统人为把它每 4096 字节切一段来管理。

### 1.2 字节寻址的物理含义

CPU 通过**地址总线**访问内存。地址总线是一组导线，每根导线表示一个二进制位。64 根线就能表示 64 位地址。

CPU 想读第 N 个字节时，在地址总线上发出数字 N，内存芯片收到后把第 N 个字节（8 个 bit）通过**数据总线**送回来。

- 你没办法只读其中 1 个 bit
- 也没办法用一个地址读到 16 个 bit
- **一个地址，硬件固定给你 8 个 bit**——这就是"按字节寻址"

> [!TIP]
>
> **为什么是 8 位为一个字节？**
> 这是历史和实用性共同决定的工程权衡。早期计算机按"字"（word）寻址（一个地址对应 12 位、36 位等），处理文本时非常浪费。IBM System/360（1964 年）率先采用 8 位字节寻址，刚好能表示一个 ASCII 字符（0~255），之后被 x86、ARM、RISC-V 等几乎所有架构沿用，成为事实标准。粒度太大（按字寻址）浪费空间，粒度太小（按位寻址）地址空间消耗太快且电路更复杂，8 位是一个实用的平衡点。

> [!TIP]
> CPU 实际上经常一次从内存取 64 位（8 字节）来提高效率，但这是缓存层面的优化。从编程模型上看，最小单位仍然是 1 字节 = 8 bit = 1 个地址。

### 1.3 地址空间的划分：为什么内存地址从 `0x80000000` 开始

RISC-V 平台（QEMU 模拟的 `virt` 机器）的硬件设计规定了**物理地址空间**的布局：

```
0x00000000 ~ 0x7FFFFFFF    外设（UART、硬盘控制器、中断控制器等）
0x80000000 ~ 0x87FFFFFF    物理内存（RAM），共 128MB
```

这个分配是**硬件工程师**设计电路板（或 QEMU 开发者写模拟器）时定下来的：

- CPU 发出地址 `0x80000000` → 硬件**地址解码电路**把请求路由到 **RAM 芯片**
- CPU 发出地址 `0x10000000` → 地址解码电路把请求路由到 **UART 串口设备**

RAM 芯片本身并不知道自己"叫" `0x80000000`。它只知道自己有 128MB 的存储。是硬件的地址解码电路决定了：CPU 地址 `0x80000000` 对应 RAM 的第 0 个字节，`0x80000001` 对应第 1 个字节。

> [!TIP]
> `0x80000000` 是 64 位地址，写完整是 `0x00000000_80000000`，高 32 位全是 0，省略不写，就像十进制 `007` 写成 `7`。

### 1.4 从硬件到 C 代码：地址值怎么传入程序

整个链条是：**硬件规定起点 → 链接器算出内核终点 → C 代码从这两个数字开始**。

**途径一：宏定义**（程序员根据硬件规格手动写的常量）

```c
// memlayout.h
#define PHYSTOP 0x88000000  // 物理内存上限
```

**途径二：链接脚本**（链接器在编译时自动计算内核镜像的结束地址）

```
// kernel/kernel.ld
. = ALIGN(0x1000);
PROVIDE(end = .);
```

C 代码通过 `extern char end[]` 引用 `end`，它的值就是内核代码和数据在内存中的结束地址。这个地址之前是内核镜像，之后才是可供分配的空闲物理内存。

然后 `kinit` 调用 `freerange(end, (void*)PHYSTOP)`，这两个地址值就传进去了。之后 `freerange` 里 `p += PGSIZE` 每次加 4096，自然产生了所有空闲页的起始地址。**没有什么魔法。**

### 1.5 xv6 物理内存的全景图

```
地址              内容
──────────────    ──────────────────────────────────────
0x80000000        [内核代码第 1 个字节]
0x80000001        [内核代码第 2 个字节]
...               ...
0x80008ABC        [内核代码最后一个字节]  ← end
0x80008ABD        [未对齐的零碎空间]
...               ...
0x80008FFF        [未对齐的零碎空间]
0x80009000        [空闲页 #1 开始，前 8 字节存 next 指针]
0x80009001        [next 指针的第 2 个字节]
...               ...
0x80009007        [next 指针的第 8 个字节]
0x80009008        [这个页剩余的 4088 字节，没人用]
...               ...
0x80009FFF        [这个页的最后一个字节]
0x8000A000        [空闲页 #2，前 8 字节又是 next 指针]
...               ...
0x87FFFFFF        [物理内存最后一个字节]  ← PHYSTOP
```

## 2. 前置知识：C 指针要点

### 2.1 指针大小与类型

- 64 位系统中，**所有指针占 8 字节**，存的是一个地址值（如 `0x80002000`）。
- 指针的**类型不影响值本身**，只决定编译器怎么解读这个地址处的内存。

### 2.2 `void *`——万能指针

- 表示"指向某个地址，但不指定类型"。
- 使用前必须强制转换成具体类型，如 `(struct run*)pa`。
- 强制转换**不改变地址值**，只是告诉编译器"按这个类型的布局来读写"。

### 2.3 `char *`——用于地址运算

C 语言中，**指针加 1 实际前进的字节数取决于指针的类型**：

```c
int *pi    = (int *)0x80009000;
pi += 1;   // pi 变成 0x80009004，前进了 sizeof(int) = 4 字节

uint64 *pu = (uint64 *)0x80009000;
pu += 1;   // pu 变成 0x80009008，前进了 sizeof(uint64) = 8 字节

char *pc   = (char *)0x80009000;
pc += 1;   // pc 变成 0x80009001，前进了 sizeof(char) = 1 字节
```

> [!TIP] 规则：
> `指针 + N` 实际前进 `N × sizeof(指针所指类型)` 个字节。

`char` 是 1 字节，所以 `char *` 的步长恰好是 1 字节，`p += 4096` 就是地址前进**正好 4096 字节**（一个页）。如果用 `int *`，`p += 4096` 就会前进 `4096 × 4 = 16384` 字节，不是我们想要的。

这就是 `freerange` 中用 `char *p` 做遍历的原因——**让地址运算的单位是字节**，精确控制步进量：

```c
char *p = (char*)PGROUNDUP((uint64)pa_start);  // p = 0x80009000
p += PGSIZE;  // p = 0x80009000 + 4096 = 0x8000A000，刚好下一个页的起始地址
```

### 2.4 `->` 运算符

`r->next` 等价于 `(*r).next`，意思是：先找到 `r` 指向的那块内存，再从中取出 `next` 字段。编译器做两件事：

1. **定位**：计算 `next` 字段在结构体中的偏移量
2. **读取**：从 `r + 偏移量` 的地址开始，读取该字段大小的字节数，按字段类型解释

用一个通用的例子说明：

```c
struct node {
    int value;           // 偏移 0，占 4 字节
    struct node *next;   // 偏移 8（对齐后），占 8 字节
};

struct node *r = (struct node *)0x80009000;
```

```
r->value :  从 0x80009000 + 0 开始，读 4 字节，解释为 int
r->next  :  从 0x80009000 + 8 开始，读 8 字节，解释为 struct node *
```

偏移量由字段在结构体中的位置决定，编译器自动计算，不需要手动算。

> [!TIP]
> 如果结构体只有一个指针字段（如后面会看到的 `struct run`），那偏移量就是 0，`r->next` 读的就是 `r` 所指地址最开头的 8 个字节。这个特性在后面第 4、5 章中会结合具体代码详细展开。

## 3. 物理地址与虚拟地址

| | 用户程序 | xv6 内核 | Linux 内核 |
|---|---|---|---|
| 使用的地址 | 虚拟地址 | 物理地址（直接映射） | 虚拟地址（带偏移的直接映射） |
| 映射关系 | 经页表翻译，程序不知道物理地址 | 虚拟地址 = 物理地址（偏移为 0） | 虚拟地址 = 物理地址 + `PAGE_OFFSET` |

xv6 内核的直接映射意味着：拿到一个物理地址值，**直接当指针用**就能读写那块物理内存。

```c
char *page = kalloc();   // 返回物理地址，如 0x80001000
page[0] = 'a';           // 直接写物理内存，因为内核虚拟地址 = 物理地址
```

## 4. 核心数据结构：侵入式链表

```c
struct run {
    struct run *next;    // 唯一字段，偏移量 0，大小 8 字节
};

struct {
    struct spinlock lock;
    struct run *freelist; // 链表头指针
} kmem;
```

`struct run` 是**链表节点的类型定义**。它只有一个字段 `next`，是一个指向 `struct run` 的指针（即指向下一个同类型节点）。这个字段的作用就是告诉编译器：当我拿到一个 `struct run *` 指针时，从这个地址开头读 8 字节，当作下一个节点的地址。

`kmem` 是全局变量，**`kmem.freelist` 存的是链表第一个节点的地址**，`kmem.lock` 是保护链表的自旋锁（防止多个 CPU 同时操作链表）。

但问题来了：要用链表管理空闲页，每个节点需要存一个 `next` 指针（8 字节）。那这个指针存在哪里？额外分配内存来存？但我们正在管理的就是内存本身，陷入了鸡生蛋的问题。

**解决办法**：空闲页反正没人在用，直接把页的**前 8 字节复用为 `next` 指针**，指向下一个空闲页的起始地址，页本身就是链表节点。

具体怎么做到的？靠**强制类型转换**把页地址和 `struct run` 关联起来：

```c
// 假设有一个空闲页，起始地址是 0x80009000
void *pa = (void *)0x80009000;

// 强制转换：告诉编译器"把这个地址值当成指向 struct run 的地址值来看待"
struct run *r = (struct run *)pa;
```

**转换前后，`r` 的值还是 `0x80009000`，没有变化**。但现在编译器认为以 `0x80009000` 这个地址为起始地址处存放着一个 `struct run` 结构体。

因为 `struct run` 只有一个字段 `next`，偏移量为 0（参见 2.4 节），所以：

- `r->next`（写入）= 往 `0x80009000` 开头的 8 字节写入一个地址值
- `r->next`（读取）= 从 `0x80009000` 开头的 8 字节读出一个地址值

**页和 `struct run` 不是两个东西**——没有单独的 `struct run` 对象被分配出来。是同一块 4096 字节的物理内存，只不过我们用 `struct run *` 的视角去看它的前 8 字节，把这 8 字节当作 `next` 指针来使用。剩余的 4088 字节在页空闲时完全不用管。

```
物理页（4096 字节）的内存布局：
┌──────────────────┬────────────────────────────┐
│ next 指针 (8B)    │ 剩余 4088 字节（不使用）    │
└──────────────────┴────────────────────────────┘

链表结构：
freelist → 页A(0x8000B000) → 页B(0x8000A000) → 页C(0x80009000) → 0
           前8字节存B的地址    前8字节存C的地址    前8字节存0
```

`freelist` 存第一个页的地址，第一个页前 8 字节存第二个页的地址，以此类推，最后一个页前 8 字节存 `0` 表示链表结束。

逐字节来看（RISC-V 小端序，低字节在低地址），页A 的前 8 字节存储 `0x8000A000`：

```
地址          内容（1 字节）    说明
0x8000B000    0x00             ┐
0x8000B001    0xA0             │
0x8000B002    0x00             │ 这 8 个字节合起来
0x8000B003    0x80             ├ = 0x00000000_8000A000
0x8000B004    0x00             │ = 下一个空闲页的地址
0x8000B005    0x00             │
0x8000B006    0x00             │
0x8000B007    0x00             ┘
0x8000B008    ...              （这个页剩余的 4088 字节，不关心）
```

## 5. 三个关键函数

### 5.1 `freerange`——初始化：把所有空闲物理页加入链表

```c
void freerange(void *pa_start, void *pa_end) {
    char *p;
    p = (char*)PGROUNDUP((uint64)pa_start); // 向上对齐到页边界
    for(; p + PGSIZE <= (char*)pa_end; p += PGSIZE)
        kfree(p);                           // 逐页加入链表
}
```

传参的时候：

- `pa_start` = `end`（内核镜像结束地址，由链接脚本 `kernel.ld` 定义）
- `pa_end` = `PHYSTOP`（物理内存上限，`0x88000000`）
- `PGROUNDUP` 宏将地址向上对齐到 4096 的整数倍（页边界）

**具体数字示例**（假设 `end = 0x80008ABC`）：

```
PGROUNDUP(0x80008ABC) = 0x80009000

第1次循环: p = 0x80009000 → kfree(0x80009000)
第2次循环: p = 0x8000A000 → kfree(0x8000A000)
第3次循环: p = 0x8000B000 → kfree(0x8000B000)
... 一直到接近 PHYSTOP
```

### 5.2 `kfree`——释放一页（头插法）

```c
void kfree(void *pa) {
    struct run *r;
    if(((uint64)pa % PGSIZE) != 0 || (char*)pa < end || (uint64)pa >= PHYSTOP)
        panic("kfree");

    memset(pa, 1, PGSIZE);         // 填垃圾值，帮助检测悬挂引用
    r = (struct run*)pa;           // 把页地址强转为 struct run *
    acquire(&kmem.lock);
    r->next = kmem.freelist;       // 新节点的 next 指向当前链表头
    kmem.freelist = r;             // 链表头更新为新节点
    release(&kmem.lock);
}
```

关键一行 `r = (struct run*)pa`：`pa` 是物理页的起始地址（如 `0x80009000`），强转成 `struct run *` 后，`r->next` 就是往这个页的前 8 字节写入/读取 `next` 指针（回顾第 4 章）。

**逐步图示**（初始 `freelist = 0`）：

```
kfree(0x80009000) 后:
  kmem.freelist = 0x80009000
  链表: 0x80009000 → 0
  （页 0x80009000 前 8 字节存的是 0，表示链表到头了）

kfree(0x8000A000) 后:
  kmem.freelist = 0x8000A000
  链表: 0x8000A000 → 0x80009000 → 0
  （页 0x8000A000 前 8 字节存的是 0x80009000）

kfree(0x8000B000) 后:
  kmem.freelist = 0x8000B000
  链表: 0x8000B000 → 0x8000A000 → 0x80009000 → 0
  （页 0x8000B000 前 8 字节存的是 0x8000A000）
```

### 5.3 `kalloc`——分配一页（摘链表头）

```c
void *kalloc(void) {
    struct run *r;
    acquire(&kmem.lock);
    r = kmem.freelist;            // 取链表头
    if(r)
        kmem.freelist = r->next;  // 链表头移到下一个节点
    release(&kmem.lock);

    if(r)
        memset((char*)r, 5, PGSIZE); // 填垃圾值
    return (void*)r;                  // 返回页地址（链表为空则返回 0）
}
```

`r->next`：从 `r` 所指页的前 8 字节读出下一个空闲页的地址，赋给 `freelist`，完成"摘头"。

**图示**（接上面的状态）：

```
kalloc() 前:

  kmem.freelist
       │
       ▼
  0x8000B000 ──→ 0x8000A000 ──→ 0x80009000 ──→ 0

kalloc() 后:

  kmem.freelist
       │
       ▼
  0x8000A000 ──→ 0x80009000 ──→ 0

  返回值: 0x8000B000（已从链表摘下，调用者可自由使用这 4096 字节）
```

## 6. 整体行为：栈（LIFO）

| 操作 | 行为 | 链表变化 |
|---|---|---|
| `kfree` | 头插 | 链表变长，新页在最前面 |
| `kalloc` | 头取 | 链表变短，取走最前面的页 |

后进先出：最后被 `kfree` 释放的页，最先被 `kalloc` 分配出去。当 `freelist = 0` 时内存耗尽，`kalloc` 返回 0。

## 7. 统计空闲内存

遍历 `freelist` 链表，数节点个数，乘以 `PGSIZE`：

```c
uint64 kfreemem(void) {
    struct run *r;
    uint64 count = 0;

    acquire(&kmem.lock);
    for(r = kmem.freelist; r; r = r->next)
        count++;
    release(&kmem.lock);

    return count * PGSIZE;
}
```
