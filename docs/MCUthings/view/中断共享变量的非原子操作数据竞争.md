# 中断共享变量的非原子操作数据竞争（Race Condition）

在嵌入式 C 语言开发中，主循环（Thread Context）与中断服务程序（ISR Context）共享全局变量时，极易因**非原子操作**引发隐蔽且致命的数据丢失问题（Lost Update）。

本文通过汇编级推演，深入剖析“读-改-写”操作在并发环境下的灾难性后果。

## 1. 核心概念剖析

在 32 位架构（如 ARM Cortex-M 或 RISC-V RV32）上，我们需要严格区分两种对全局变量的操作：

### 1.1 纯读/纯写 (Read-Only / Write-Only) - 天然原子性
例如 `ticks = system_ticks;` 或 `system_ticks = 0;`。
只要变量是 32 位且自然对齐的，编译器通常会将其翻译为**单条**汇编指令（如 RISC-V 的 `lw` 或 `sw`）。
单条机器指令在执行期间**物理上不可被中断打断**，因此不会读到“半旧半新”的撕裂值（Torn Read）。此类操作**无需关中断**（前提是变量必须用 `volatile` 修饰以防止寄存器缓存）。

### 1.2 读-改-写 (Read-Modify-Write) - 非原子性
例如 `system_ticks += inc_ms;`、`flag |= 0x01;` 或 `count++;`。
这类操作在底层会被编译器拆分为**三条独立的汇编指令**：
1.  **Load (读)**：从内存读入 CPU 寄存器。
2.  **ALU (改)**：在寄存器中进行算术或逻辑运算。
3.  **Store (写)**：将计算结果从寄存器写回内存。

中断随时可能在这三条指令的**间隙**发生，从而引发数据竞争。

## 2. 灾难推演：Lost Update (数据丢失)

假设全局变量 `system_ticks` 当前值为 `1000`。
*   **主循环（Thread）** 执行：`system_ticks += 2;`
*   **定时器中断（ISR）** 执行：`system_ticks += 5;`

预期结果应为 `1007`。真实的时间线推演如下：

| 步骤 | 执行上下文 | 汇编指令 (RISC-V) | 操作说明 | `system_ticks` (内存值) |
| :--- | :--- | :--- | :--- | :--- |
| 1 | **主循环** | `lw a4, 0(a5)` | **[读]** 将 `1000` 读入寄存器 `a4` | 1000 |
| 2 | **主循环** | `add a0, a0, a4` | **[改]** 寄存器内计算: `1000 + 2 = 1002` | 1000 |
| **💥** | **<中断发生>** | | 硬件强行打断主循环，跳入 ISR | 1000 |
| 3 | **ISR** | `lw a4, 0(a5)` | **[读]** 将内存中的 `1000` 读入 ISR 寄存器 | 1000 |
| 4 | **ISR** | `add a0, a0, a4` | **[改]** ISR 寄存器内计算: `1000 + 5 = 1005` | 1000 |
| 5 | **ISR** | `sw a0, 0(a5)` | **[写]** 将 `1005` 写回内存。ISR 返回 | **1005** (被 ISR 更新) |
| 6 | **主循环** | `sw a0, 0(a5)` | **[写]** 主循环恢复现场，将之前算好的 `1002` 写回内存！ | **1002** (灾难发生) |

### 灾难性后果
*   **数据被覆盖**：中断里辛辛苦苦累加的 5 个 tick **凭空消失了**，被主循环过期的计算结果无情覆盖。
*   **系统级影响**：会导致系统时钟变慢、超时机制错乱（如按键长按判定失败、蓝牙协议栈超时断开等）。

## 3. 防御策略 (Best Practices)

当必须在非中断上下文中对共享变量进行“读-改-写”时，必须采用以下策略之一进行保护：

### 3.1 关全局中断 (最稳妥、最通用)
在操作前后插入关/开中断的保护墙（临界区）。
```c
extern volatile uint32_t system_ticks;

void add_ticks_safely(uint32_t inc_ms) {
    // 1. 保存当前中断状态并关闭全局中断
    uint32_t primask = __get_PRIMASK(); // (ARM 示例)
    __disable_irq();

    // 2. 临界区：安全地进行读-改-写
    system_ticks += inc_ms;

    // 3. 恢复之前的中断状态
    __set_PRIMASK(primask);
}
```

### 3.2 硬件原子指令 (优雅但依赖架构)
如果目标芯片的指令集支持原子扩展（如 RISC-V 的 `A` 扩展，或 ARM 的 `LDREX/STREX` 独占访问机制），可以直接使用编译器内置的原子函数，无需关中断。
```c
// 使用 GCC 内置的原子加法函数
__atomic_add_fetch(&system_ticks, inc_ms, __ATOMIC_RELAXED);
```
*(注：对于小型 MCU，关中断通常是兼容性最好、最不易出错的做法。)*
