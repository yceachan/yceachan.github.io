# epoll LT 与 ET 深度解析

> [!note]
> **Ref:**
> - man pages: `epoll(7)`
> - 内核源码: `fs/eventpoll.c`
> - 相关笔记: [`06-multiplex-compare.md`](./06-multiplex-compare.md)

## 1. 核心概念

epoll 的两种模式定义了内核何时认为一个文件描述符(fd)是"就绪"的并通知用户态。

### 1.1 水平触发 (Level-Triggered, LT)
**默认模式**。只要 fd 处于"就绪状态",`epoll_wait` 就会不断通知。

*   **语义**: "缓冲区还有数据/空间吗?" → "有" → 报告。
*   **特点**: 容错性高,允许部分读取。
*   **内核逻辑**: 当 `epoll_wait` 返回事件给用户后,如果该 fd 仍满足触发条件(缓冲区未空),内核会将该 `epitem` 重新放回就绪链表 `rdllist`,导致下一次调用立即返回。

### 1.2 边沿触发 (Edge-Triggered, ET)
通过 `EPOLLET` 标志启用。仅在 fd 状态发生 **变化** 时通知一次。

*   **语义**: "有新数据到了吗?" → "有" → 报告一次。
*   **特点**: 极其高效,减少内核/用户态切换,但对程序员要求极高。
*   **内核逻辑**: 当 `epoll_wait` 返回事件后,内核会从 `rdllist` 中移除该项,且**不会**自动放回。除非驱动层再次调用 `wake_up` (表示有新数据到达),否则该 fd 不会再出现在 `rdllist` 中。

## 2. 行为对比：2KB 数据场景

假设 socket 接收缓冲区到了 2KB 数据:

| 步骤 | 动作 | LT 模式行为 | ET 模式行为 |
| :--- | :--- | :--- | :--- |
| 1 | 数据到达 | `epoll_wait` 返回 | `epoll_wait` 返回 |
| 2 | 应用 read(1KB) | 缓冲区还剩 1KB | 缓冲区还剩 1KB |
| 3 | 调用 `epoll_wait` | **立即返回** (因为还有 1KB) | **阻塞** (直到下一波新数据) |

**结论**: ET 模式下如果不一次性读完(直到 `EAGAIN`),剩下的数据会"滞留"在缓冲区,直到下一次新事件触发。

## 3. ET 模式的两条铁律

使用 ET 模式必须严格遵守以下两条准则,否则会导致死锁或程序挂起:

### 3.1 必须使用非阻塞 IO (O_NONBLOCK)
*   **原因**: ET 要求循环读取直到缓冲区空。如果使用阻塞 read,最后一次读取(当缓冲区已空时)会使进程卡死,整个事件循环无法继续。
*   **后果**: 如果不设非阻塞,程序会死在 `read()` 调用上。

### 3.2 必须循环读取直到返回 EAGAIN
*   **原因**: ET 只在状态变化时通知。如果只读一次而没读尽,内核不会再发通知。
*   **代码模板**:
    ```c
    while (1) {
        ssize_t n = read(fd, buf, sizeof(buf));
        if (n > 0) {
            // 处理数据
        } else if (n == 0) {
            // 对端关闭
            break;
        } else {
            if (errno == EAGAIN || errno == EWOULDBLOCK) {
                // 读完了,优雅退出循环
                break;
            }
            // 真正的错误
            perror("read");
            break;
        }
    }
    ```

## 4. 为什么 ET 更快？

1.  **减少系统调用次数**: 只要处理逻辑正确,一个事件通知就能处理掉所有积压数据。
2.  **减少就绪链表维护开销**: 内核不需要频繁地将 `epitem` 在红黑树和就绪链表之间搬运。
3.  **配合多线程更高效**: 配合 `EPOLLONESHOT`,可以确保一个 socket 在某一时刻只被一个 worker 线程处理,避免竞态。

---

## 5. 惊群效应 (Thundering Herd) 与解决方法

当多个进程/线程同时 `epoll_wait` 监听同一个 listen socket 时,一个新连接到来可能唤醒所有等待者,但只有一个能 `accept` 成功。

*   **LT 模式**: 容易产生惊群。
*   **ET 模式**: 依然可能产生惊群,但较 LT 稍好。
*   **现代方案 (Linux 4.5+)**: 使用 `EPOLLEXCLUSIVE` 标志,内核保证只唤醒一个进程。

---

## 6. 总结建议

*   **选 LT**: 逻辑简单,适合大多数中小规模应用,或对实时性要求不高、处理逻辑较重的场景。
*   **选 ET**: 需要极致并发性能,或处理的是大量短小请求。**必须配合非阻塞 IO 和状态机编程**。
