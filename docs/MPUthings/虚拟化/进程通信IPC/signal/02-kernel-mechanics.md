# 内核信号分流机制：命运的抉择

`kill` 命令的本质是 `send_signal`。当信号到达内核，并准备从内核态返回用户态时，内核会在 `do_signal()` 中对进程进行**命运的分流**。

## 1. 命运 A：拦截并存活 (Catching)

如果进程提前通过 `sigaction` 注册了处理函数：
1. **栈帧注入**：内核在用户栈压入“信号栈帧”，保存当前寄存器。
2. **PC 指针重定向**：返回用户态时，强行让 PC 跳到注册的回调地址（如 `handle_sigusr1`）。
3. **返回流程**：回调执行完，触发 `rt_sigreturn` 系统调用，内核将 PC 拨回原中断点，进程**继续运行**。

## 2. 命运 B：强制资源剥夺 (Abort)

如果进程未注册（`SIG_DFL`）或信号属于致命信号：
1. **控制权拦截**：内核拒绝交还 PC 指针。
2. **清理现场 (`do_group_exit`)**：立即在内核上下文中关闭 FD、释放内存页、清理页表。
3. **僵尸化**：进程化为 `EXIT_ZOMBIE`，只保留一个外壳等待父进程 `wait()`。

## 3. 为什么 `kill -9` (SIGKILL) 无法逃脱？

`SIGKILL` (9) 和 `SIGSTOP` (19) 在内核层级被硬编码为**不可捕获、不可忽略、不可阻塞**。
- 内核提取信号时直接跳过分流逻辑，无条件进入 `do_group_exit()`。
- 它是内核赋予管理员的绝对控制权。

## 4. 僵尸进程的成因与危害

当进程进入“命运 B”被杀掉后，如果其父进程没有及时调用 `wait()` 或 `waitpid()`，该进程的 PID 就会一直残留在内核 `task` 列表中，占用系统资源。
- **解决方法**：捕获 `SIGCHLD` 信号，在处理函数中异步调用 `waitpid` 进行清理。
